# CapsLock RS 开发计划

## 1. 项目定位

CapsLock RS 的核心目标是将 `CapsLock` 变成一个可配置的键盘修饰层，使用户能够建立如下映射：

```text
CapsLock 起始组合键
    -> 单个目标按键
    -> 参数化内置函数
    -> 目标组合键
```

项目优先保证输入稳定性、配置自由度和可理解性。功能数量不是目标，用户能够准确表达自己的键盘习惯才是目标。

## 2. 设计原则

### 2.1 CapsLock 是修饰键

核心模型应接近 `Ctrl`、`Alt` 和 `Shift`，而不是把 CapsLock 变成打开搜索框或复杂工作台的入口。

### 2.2 INI 是持久化格式

GUI 和手工编辑共同使用 INI 文件。GUI 不是另一套配置系统，只是 INI 的可视化编辑器。

### 2.3 配置应宽容，但行为应明确

- 键名和函数名可支持合理别名及大小写差异。
- 未知配置应记录位置明确的警告，尽可能跳过单条错误映射。
- 冲突映射、无效参数和危险动作必须明确提示。
- 配置错误不能导致钩子线程崩溃或键盘失去响应。

### 2.4 输入核心与界面分离

键盘钩子、映射解析、动作执行、GUI 和 i18n 分层实现。无论 GUI 是否运行，后台输入核心都应能独立工作。

### 2.5 最小权限

默认使用普通用户权限。管理员模式等敏感运行方式必须由用户显式开启，并保持可审计的配置和日志。

## 3. 明确不开发的功能

以下功能不进入核心路线：

- QBar、文件搜索和应用搜索。
- 翻译和其他在线 API 服务。
- 计算器、表达式求值和计算草稿纸。
- 多剪贴板或剪贴板历史管理。
- 综合窗口管理、桌面管理或鼠标自动化套件。
- 按键映射触发启动程序、打开 URL 或执行脚本的外部动作。
- 在主进程内执行任意代码的通用脚本环境。

后续需求应先判断是否直接服务于键位映射、按键模拟或配置管理。不能满足这一条件的功能默认拒绝进入核心项目。

## 4. 当前基线：里程碑 1

- CapsLock 功能层和基础状态机。
- 上、下、左、右方向键模拟。
- Enter、Backspace 和 Delete。
- `moveUp(5)`、`moveDown(5)` 等次数参数。
- `moveWordLeft` 和 `moveWordRight`。
- CapsLock+ 风格 INI 解析。
- 后台运行、托盘菜单、单实例和日志。

## 5. 里程碑 2：通用键位与组合键模型

状态：已完成。

目标：将当前针对少数动作的映射实现，升级为通用、可扩展的键盘表达系统。

### 输入组合键

- 覆盖字母、数字、方向键、功能键、编辑键、标点键和常用媒体键。
- 支持 `Ctrl`、`Alt`、`Shift`、`Win` 及其左右版本。
- 支持多修饰键组合，例如 `caps_ctrl_h`、`caps_lalt_shift_j`。
- 对修饰键顺序进行标准化，避免同一组合出现多种内部表示。
- 明确区分物理键、虚拟键和受键盘布局影响的字符键。

### 输出动作

动作模型至少分为三类：

1. 单个目标按键，例如输出 `F5`、`Home` 或媒体播放键。
2. 内置目标函数，例如 `moveUp(5)`、`selectUp(5)` 或 `deleteWord`。
3. 目标组合键，例如输出 `Ctrl+C`、`Ctrl+Shift+Left`。

当前 INI 语法继续接近 CapsLock+ 的函数调用风格，并已实现以下三类动作：

```ini
[Keys]
caps_h=keyFunc_moveLeft
caps_u=keyFunc_selectUp(5)
caps_r=keyTarget_f5
caps_c=keyCombo_ctrl_c
caps_lalt_d=keyFunc_moveWordRight
```

### 内置编辑动作

已增加：

- `selectLeft(n)`、`selectRight(n)`、`selectUp(n)`、`selectDown(n)`。
- `selectWordLeft(n)`、`selectWordRight(n)`。
- `home`、`end`、`pageUp`、`pageDown`。
- `deleteWord` 和 `forwardDeleteWord`。
- 可复用的按键按下、释放和点击执行器。

### 稳定性要求

- 任何输出组合键结束后都必须正确释放临时修饰键。
- 程序生成的输入不能再次触发自身映射。
- 长按、系统自动重复和快速交替按键不能造成卡键。
- 未识别映射只禁用对应条目，不应使整个配置失效。
- 增加解析器、动作转换和修饰键生命周期测试。

### 验收标准

- 常用键位无需重新编译即可通过 INI 映射。
- 用户可以配置 CapsLock 起始的多修饰键组合。
- 用户可以选择输出单键、内置函数或组合键。
- 当前已经存在的 CapsLock+ 风格配置继续工作。

## 6. 里程碑 3：i18n 基础

状态：部分完成。

目标：在 GUI 大量出现文本之前建立本地化基础，至少完整支持简体中文。

- 增加 `language = system|zh-CN|en-US` 配置。
- 首批提供 `zh-CN` 和 `en-US` 资源，简体中文必须完整可用。
- 跟随系统语言时，无法匹配的语言回退到英文或项目设定的默认语言。
- 托盘菜单、消息框、配置校验结果和 GUI 文本统一通过翻译键获取。
- 禁止在界面模块中继续散落硬编码英文字符串。
- 翻译资源缺少键时使用稳定回退，并在开发日志中报告。
- 保持 Windows UTF-16 边界和 INI UTF-8/UTF-16 兼容。

日志中的机器可检索字段可保持稳定英文；面向用户显示的错误摘要应支持中文。

当前已经完成的范围：

- `language = system|zh-CN|en-US` 配置解析、序列化和设置模型传递。
- `zh-CN` 与 `en-US` 翻译表，托盘菜单和消息框错误摘要已通过翻译键获取。
- 缺失翻译键时先回退到英文，仍缺失时使用稳定占位文本，并写入日志。
- 已增加测试，检查中英文翻译 key 集合一致、回退行为稳定且会记录缺失 key。

当前仍未完成的范围：

- GUI 设置页文本已接入资源层，但完整配置流程仍需继续通过实际编辑、保存和语言切换回归验证。
- 配置解析错误的底层详情仍以英文和机器可检索日志为主，后续 GUI 校验页需要再补用户友好的本地化说明。

### 验收标准

- 中文 Windows 上首次运行能够显示完整中文托盘菜单和错误信息。
- 用户可手动切换中文、英文或跟随系统，并在重载后生效。
- 新增界面文本必须通过 i18n 资源提供。

## 7. 里程碑 4：GUI 配置页

状态：部分完成。

当前已经完成的范围：

- 原生 Win32 设置页可从托盘打开，并复用当前单实例后台进程。
- 基础页面可编辑启用状态、开机启动、管理员模式、托盘图标、CapsLock 单击行为和界面语言。
- 映射编辑器的代码路径已实现新增、更新、删除 CapsLock 输入组合，并配置内置函数、目标单键和目标组合键；真实窗口流程仍需继续手工回归。
- 源组合键和目标组合键的编辑控件已支持下拉选择和按键监听录入；真实窗口下的录入体验仍需继续手工回归。
- 保存路径代码层面复用 `SettingsModel`、`Config` 解析/序列化、原子保存和自动重载，不直接修改钩子内部状态。
- GUI 保存会从当前配置应用可编辑字段，已识别的 `general`、`ui` 字段和三类 `[Keys]` 映射不会无故丢失；注释、未知键、原始顺序和格式目前不会保留。
- 映射列表已显示正常、重复映射、非法输入组合、未知动作或配置错误等行级状态，状态来源复用 `SettingsModel` 和 `ConfigValidation`。
- 当前编辑行已显示保存前将写入的 INI 表达式预览；预览成功路径复用 `KeyBindingRow` 到 `Config` 的序列化结果，非法内容显示本地化错误摘要。

当前仍未完成的范围：

- 行级状态已接入映射列表，但完整编辑期校验体验仍需继续回归。
- 中文和英文资源已覆盖当前设置页文本，但真实窗口下的完整双语流程仍需继续做实际回归。

目标：让不熟悉 INI 的用户也能安全地新增和修改映射，同时保留手工配置能力。

### 基础页面

- 启用或暂停 CapsLock 层。
- 开机启动。
- 管理员模式。
- 托盘图标显示。
- CapsLock 单击行为。
- 界面语言。
- 当前配置路径和日志路径。

### 映射编辑器

映射表至少展示：

| 字段 | 内容 |
| --- | --- |
| 输入组合 | 以 CapsLock 开始的源按键组合 |
| 动作类型 | 目标按键、内置函数或目标组合键 |
| 动作内容 | 键名、函数名、参数或组合键 |
| 状态 | 正常、重复映射、非法输入组合、未知动作或配置错误 |

新增或编辑映射时应支持：

- 直接按键录入源组合键。
- 从列表选择动作类型。
- 根据动作类型显示对应的键位、函数和参数控件。
- 检查重复映射、非法按键和超出范围的参数。
- 保存前预览将写入的 INI 表达式。
- 保存后原子替换配置文件并自动重载。
- 写入失败时保留原配置，并显示可理解的错误。

### 架构要求

- GUI 只操作独立的设置模型，不直接修改键盘钩子内部状态。
- 配置解析与序列化是唯一规则来源，GUI 不复制一套语法规则。
- GUI 关闭后后台程序继续运行。
- 设置页应能从托盘打开，并复用当前单实例进程。

### 验收标准

- 用户可以只通过 GUI 新建一个 CapsLock 组合键映射。
- 三种目标类型均可配置、保存、重载和删除。
- GUI 与手工编辑同一份 INI 后，不会无故丢失可识别配置。
- 中文和英文界面均可完整完成配置流程。

当前验收对照：

- “用户可以只通过 GUI 新建一个 CapsLock 组合键映射”：代码路径已实现，仍需继续积累真实环境手工验证。
- “三种目标类型均可配置、保存、重载和删除”：代码路径和单元测试已覆盖 keyFunc、keyTarget、keyCombo 的新增、更新、删除、INI 预览与 INI 往返；真实窗口保存后运行时重载仍需手工回归。
- “GUI 与手工编辑同一份 INI 后，不会无故丢失可识别配置”：部分满足；已识别字段和映射会保留，注释、未知键和原始格式不保留。
- “中文和英文界面均可完整完成配置流程”：部分完成；文本资源已覆盖当前流程，但还需要更系统的双语手工回归。

## 9. 开发顺序

建议按以下顺序推进：

1. 重构通用键位、修饰键和动作数据模型。
2. 完成目标单键、组合键和选择类内置函数。
3. 建立 i18n 资源层，并先替换托盘与消息框文本。
4. 收尾 GUI 的基础设置和映射编辑器（当前部分完成）。
5. 完成许可证、CI、发布包和兼容性验证。

在每个里程碑完成前，不提前引入后续阶段的大型依赖。输入核心和配置格式的稳定性始终优先于界面数量或动作数量。
